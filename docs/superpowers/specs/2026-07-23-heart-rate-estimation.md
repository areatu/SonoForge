# Спецификация: Оценка ЧСС по видеоэхокардиограмме

**Дата:** 2026-07-23
**Статус:** Реализовано (базовый набор)
**Модуль:** `src/echo_personal_tool/domain/services/heart_rate.py`

---

## 1. Постановка задачи

Оценить частоту сердечных сокращений (ЧСС) по cine-записи эхокардиограммы в B-режиме. Результат используется для:
- Отображения в статус-баре / отчёте
- Нормализации временно́х интервалов (ejection time, isovolumic time)
- Калибровки M-mode измерений

**Ограничения:**
- Анализ по требованию (кнопка в UI), не автоматический
- Фоновый поток, не блокирует воспроизведение
- Работа на старых системах: < 2 сек на 30-кадровый cine

## 2. Реализованные методы

### 2.1. Optical Flow (оптический поток)

**Принцип:** Измерение движения пикселей между кадрами. Колебания стенок LV создают периодический сигнал в оптическом потоке.

**Алгоритм:**
1. Преобразование кадров в градации серого
2. `cv2.calcOpticalFlowFarneback` между соседними кадрами (pyr_scale=0.5, levels=3, winsize=15, iterations=3)
3. Вычисление magnitude потока `sqrt(vx² + vy²)` в ROI (по умолчанию: центральные 60% кадра)
4. Интегрирование magnitude по ROI → 1D сигнал `flow(t)`
5. Bandpass фильтр Butterworth 4-го порядка: 40–200 BPM (0.67–3.33 Hz)
6. Автокорреляция через FFT → поиск доминантного периода
7. `BPM = 60 × fps / peak_lag`

**Параметры:**
| Параметр | Значение | Обоснование |
|----------|----------|-------------|
| ROI | Центральные 60% | LV обычно в центре кадра |
| Bandpass | 40–200 BPM | Физиологический диапазон ЧСС |
| pyr_scale | 0.5 | Стандарт для эхо |
| winsize | 15 | Баланс точности/скорости |
| min_correlation | 0.3 | Порог уверенности |

**Стоимость:** ~1-2 мс на кадр (100×100 ROI). Полный 30-кадровый cine: < 0.1 сек.

**Преимущества:**
- Не требует контура или сегментации
- Работает на любом видео без ONNX
- Мгновенное выполнение

**Ограничения:**
- Чувствителен к движению камеры / диафрагмы
- Точность снижается при плохом качестве видео
- ROI должен содержать движущиеся структуры (стенки LV)

---

### 2.2. Area-Time (площадь-время)

**Принцип:** Площадь полости LV колеблется с ЧСС. Минимум площади = систола (ES), максимум = диастола (ED). Интервал ES→ES стабильнее ED→ED.

**Алгоритм:**
1. На вход: список площадей контуров `area[i]` (мм²) для выбранных кадров
2. Поиск локальных минимумов (ES frames): `area[i] < area[i-1] AND area[i] < area[i+1]`
3. Вычисление интервалов ES→ES: `interval_ms = (ES[i+1] - ES[i]) × (1000 / fps)`
4. Медiana интервалов → `BPM = 60000 / median_interval_ms`
5. Уверенность: `confidence = 1 - CV(interval)` (коэффициент вариации)

**Входные данные:**
- **Dense:** контуры на каждом кадре (точнее, но дороже)
- **Sparse:** контуры на 5-10 кадрах через равные интервалы (быстрее)
- **Существующие контуры:** если на кадре уже есть контур — используем его

**Стоимость:** Вычисление polygon area: O(n) на кадр. Поиск минимумов: O(n). Пренебрежимо мало.

**Преимущества:**
- Клинически обоснованный метод (аналог M-mode)
- Стабильнее optical flow (менее чувствителен к шуму)
- Использует уже существующие контуры

**Ограничения:**
- Требует контуры на нескольких кадрах
- При sparse sampling точность зависит от количества точек
- Не работает без сегментации LV

## 3. Неиспользованные методы (рассмотрены, отклонены)

### 3.1. Template Matching

**Принцип:** Взять ROI вокруг LV на первом кадре, искать этот шаблон на последующих кадрах (normalized cross-correlation).

**Почему отклонён:**
- Чувствителен к изменению масштаба (сердце меняет размер)
- Чувствителен к повороту
- ROI на первом кадре может не совпадать с LV на других кадрах
- Не даёт прямой сигнал ЧСС — требует дополнительной обработки

### 3.2. Gradient Oscillation (градиентная осцилляция)

**Принцип:** На каждой строке измерить суммарную градиентную энергию вдоль стенок LV. Колебания градиента → ЧСС.

**Почему отклонён:**
- Низкая точность (градиенты зависят от УЗИ-настроек: gain, TGC, depth)
- Требует предварительного определения ROI стенок
- Чувствителен к артефактам (dropout, clutter)
- Optical flow даёт тот же результат лучше

### 3.3. ECG-синхронизация (априори)

**Принцип:** Если доступен ECG-сигнал (в DICOM-тегах или отдельном канале), ЧСС определяется напрямую по R-пикам.

**Почему не реализовано (пока):**
- Не все DICOM-файлы содержат ECG
- Требует парсинга SequenceOfECGWaveform (0018,6040)
- Не работает для MP4/нестандартных форматов
- **Рекомендация:** реализовать как fallback когда ECG доступен

### 3.4. Deep Learning (RNN/TCN)

**Принцип:** Обученная модель (Recurrent Neural Network / Temporal Convolutional Network) предсказывает ЧСС напрямую из последовательности кадров.

**Почему отклонён:**
- Требует обучения на размеченных данных (Cine + ECG)
- Добавляет ~100 MB к размеру приложения
- ONNX inference на последовательности: ~50-100 мс (приемлемо, но избыточно)
- Optical flow + autocorrelation дают сопоставимую точность без обучения

### 3.5. Speckle Tracking (斑点追踪)

**Принцип:** Отслеживание эхогенных斑点 (speckle patterns) в миокарде для определения механических событий (начало/конец систолы).

**Почему отклонён:**
- Это STE (Speckle Tracking Echocardiography) — уже реализовано в проекте
- STE даёт strain/strain rate, не ЧСС напрямую
- Вычислительно дорого (3D tracking)
- Избыточно для простой оценки ЧСС

## 4. Архитектура

```
HeartRateWorker (QRunnable, фоновый поток)
    │
    ├── Метод 1: Optical Flow
    │   ├── Загрузка кадров из DICOM/cine
    │   ├── cv2.calcOpticalFlowFarneback
    │   ├── Bandpass filter + autocorrelation
    │   └── HeartRateResult
    │
    ├── Метод 2: Area-Time
    │   ├── Получение контуров (из stored_contours)
    │   ├── polygon_area для каждого кадра
    │   ├── Local minima (ES) → ES→ES intervals
    │   └── HeartRateResult
    │
    └── Combined (будущее)
        ├── Optical Flow (primary)
        ├── Area-Time (validation)
        └── Взвешенное среднее
```
